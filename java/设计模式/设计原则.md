```
注意：耦合性，内聚性，可维护性，可拓展性，重用性，灵活性

1.单一职责原则：一个类只负责一项职责。
	解决方式--拆分，根据功能进行拆分多个类，多个类可以实现同一接口，并重载接口方法。
	（缺点：开销比较大，若是功能比较简单，可以在函数级别实现单一职责原则。）
	
	降低类的复杂度（提高内聚性，减少耦合度）
	提高类的可读性，可维护性
	降低变更引起的风险

2.接口隔离原则：客户端不应该依赖它不需要的类，即一个类对另一个类的依赖应该建立在最小接口上。
	依赖关系是一个类A的函数调用另一个类B的方法。
	对于实现接口的类，如果它被调用的方法不是整个接口的方法时，应该将接口拆分成需要被调用的接口的组合。
	一个类不应该实现没有被调用的方法。尽量保证接口中的每个方法都被被实现
	
	降低了耦合性，提高被依赖类的内聚性
	
3.依赖倒转原则：高层模块不应该依赖底层模块（高层不应该调用低层，应该调用低层的抽象），二者都应该依赖其抽象；
	抽象不应该依赖细节，细节应该依赖抽象，抽象指接口和抽象类，细节指具体实现类。高层指的是抽象，低层指的是细节。

	接口和抽象类的目的是指定规范，不涉及具体操作
	不要依赖具体实现，要依赖抽象

	依赖传递的三种方式：
	（1）接口传递 public void operator(Interface i)[聚合]
	（2）构造方法传递 class A   { public A(Interface i)}[组合]
	（3）setter方式传递 
	
	低层模块都要有抽象类或者接口，不要孤零零的一个实现类
	变量的声明类型尽量是抽象类或者接口
	继承时循序里氏替换原则
	
	提高了可拓展性，可维护性
	
4.里氏替换：所有引用基类的地方必须能够透明地使用其子类的对象
	使用继承时。要遵循里氏替换原则，在子类中不要重写父类方法。
	其实要尽量减少继承，继承会使耦合性增强，可以提取子类和父类的方法构成接口，去实现接口，若想要调用其父类方法，可用使用组合，聚合和依赖。
	在定义中使用父类，在实现上使用子类
	
	
5.开闭原则： 一个软件实体（类，函数，模块）对拓展开放，对修改关闭。用抽象构建框架，用实现拓展细节。 
	尽量增加方法，不是修改方法。
	只增加功能，可用使用接口来占位，传入具体实现

6.迪米特法则（最少直到原则）：一个类对自己依赖的类知道的越少越好。
	只需知道如何调用，不需知道如何实现。
	直接朋友：成员变量，方法参数，方法返回值
	出现在局部变量的类不是直接朋友，要减少。
	降低类与类之间的的耦合

7.组合复用原则：使用合成/聚合的方式，而不是继承 


设计原则核心思想：
找出应用中可能变化的地方，将它们独立出来
针对接口编程，不是针对实现编程
为交互对象的松耦合设计
```
