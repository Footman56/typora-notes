### 缓存

读路径：当应用需要查询数据时，首先检查缓存（如 Redis）中是否存在结果。如果命中，则直接返回；如果未命中，则执行数据库查询（可能包含应用层的 J0IN），然后将查询结果写入缓存，最后再返回给应用。
写路径：当底层的业务数据（如用户信息）发生变更时，为了保证数据一致性，最简单的策略是直接**淘汰（删除）**缓存中相关的条目。当
下次再有读取请求时，由于缓存未命中，它会自然地从数据库加载最新数据并重新填充缓存。

总结：

•核心在线业务，追求高性能查询：数据冗余是首选。
• 复杂报表和数据分析：数据仓库是唯一正确的方向。
•存量系统改造，希望对业务代码影响最小：可以评估引入中间件。
•高频读取、非核心数据，或作为其他方案的补充：缓存结果是强大的加速利器。
•临时低频的后台查询：应用层整合作为一种简单快速的方案，偶尔也能派上用场。

## 分布式事务

### 基于XA协议的两阶段提交

- **原理**：
  1. **准备阶段**：事务协调者询问所有参与者能否提交，参与者锁定资源并反馈Yes。
  2. **提交阶段**：如果都返回Yes，协调者发送Commit指令；如果有人返回No，协调者发送Rollback指令。
- **优点**：严格遵循ACID，强一致性。
- **缺点**：
  - **性能差**：需要锁定资源，直到第二阶段结束，高并发下数据库连接容易成为瓶颈。
  - **单点问题**：协调者宕机会导致事务阻塞。
  - **同步阻塞**：参与者等待命令时处于阻塞状态。

### 3PC（三阶段提交）

- 1. **CanCommit**：询问参与者是否具备执行条件（不锁定资源）。
  2. **PreCommit**：预执行操作，锁定资源。
  3. **DoCommit**：最终提交。

### TCC

- **原理**：要求每个参与方都实现三个接口：
  - **Try**：完成业务检查，预留资源（如冻结库存）。
  - **Confirm**：真正执行业务，使用Try阶段预留的资源（注意Confirm需幂等）。
  - **Cancel**：释放Try阶段预留的资源（注意Cancel需幂等）。

- **优点**：
  - 性能比2PC高很多，没有全局锁。
  - 解决了跨数据库、跨服务的分布式事务问题。
- **缺点**：
  - 开发成本高，每个操作都需要写三段逻辑。
  - 侵入性强，业务逻辑与事务耦合。

### 可靠消息

RocketMQ

- **原理**：
  1. 上游服务（如订单服务）在本地事务中，将业务数据和消息同时写入数据库。
  2. 通过一个定时任务或消息中间件的机制，确保消息一定能发送到MQ。
  3. 下游服务（如积分服务）消费消息，执行本地业务（需要保证幂等性）。
  4. 下游处理成功后，异步告知上游或MQ删除消息。
- **优点**：实现了高性能的最终一致性，解耦程度高。
- **缺点**：业务系统需要配合实现消息确认和重试机制。

### saga

Saga是一种长活事务的解决方案，将一个分布式事务拆分为一系列本地事务，每个本地事务都有一个对应的补偿操作。

- **T1 -> T2 -> T3**：正向执行。
- 如果T3失败了，则执行补偿操作 **C2 -> C1**。

### seata 

- **AT模式（Automatic Transaction）**：
  - 这是Seata主推的模式，是对2PC的优化。
  - **原理**：在业务SQL执行前，Seata拦截SQL，生成执行前的镜像（Before Image）和执行后的镜像（After Image），存入undo_log表。如果全局事务需要回滚，就根据镜像数据还原。
  - **优点**：对业务代码侵入小（只需加`@GlobalTransactional`注解），性能比传统XA高。
  - **缺点**：需要额外的数据库表（undo_log）来存放回滚日志。
- **TCC模式**：Seata也支持TCC模式的开发框架。



**业务是否允许短暂的不一致？**

- **不允许（强一致性）**：数据量不大，选 **2PC**；数据量大，选 **Seata AT模式**。
- **允许（最终一致性）**：选 **TCC**、**可靠消息** 或 **SAGA**。

# 水平分表

分片键和分片策略很关键

## 确定分片策略

| 策略类型     | 实现方式                                                     | 适用场景                                           | 优点                                   | 缺点                                                 |
| :----------- | :----------------------------------------------------------- | :------------------------------------------------- | :------------------------------------- | :--------------------------------------------------- |
| **范围分片** | 按数据范围划分，如 `user_id 1-100万` 在表A，`100万-200万`在表B。 | 数据有时序性，如**订单表按时间、日志表按日期**。   | 范围查询高效，扩容简单（直接加新表）。 | 容易产生**热点数据**（最新数据集中在一张表）。       |
| **哈希分片** | 通过哈希函数计算分片位置，如 `target_table = hash(user_id) % N`。 | 数据量大，追求**负载均衡**，如**用户表、订单表**。 | 数据分布均匀，写入压力分散。           | 扩容时需要**迁移大量数据**（翻倍扩容可缓解）。       |
| **目录分片** | 维护一个独立的**路由表**，记录每条数据所在的分片。           | 分片规则复杂、灵活多变，如**多租户SaaS系统**。     | 灵活性最高，不受分片键限制。           | 多一次查询，**性能有损耗**，路由表本身可能成为瓶颈。 |

## 数据库架构调整

- **创建子表**：根据你的分片策略（比如拆分成16张表），在数据库（或不同数据库实例）中创建结构完全相同的新表。例如：`orders_0`, `orders_1`, ..., `orders_15` 。
- **表结构设计**：子表的结构通常与原表一致，但需要特别注意**主键**。因为数据被分散了，不能再使用单表的自增ID，否则会重复。你需要提前规划好**全局唯一ID**的生成方案 ：
  - **雪花算法（Snowflake Algorithm）**：目前最主流的方式，生成一个64位的长整型ID，包含时间戳、机器码、序列号等信息，保证了全局唯一且趋势递增。
  - **数据库分段**：用一个中心数据库专门分发ID段，应用拿到一段ID后，再在本地分配。
  - **UUID**：最简单，但作为主键时太长且无序，会导致索引性能严重下降，**生产环境慎用**。

## 数据迁移

- **全量迁移**：编写脚本，通过 `INSERT INTO ... SELECT ...` 语句，将原大表的数据按照新规则分批、分片地插入到对应的子表中 。

  sql

  ```
  -- 示例：将原订单表中 user_id 取模为0的数据迁移到 orders_0 表
  INSERT INTO orders_0 (order_id, user_id, amount)
  SELECT order_id, user_id, amount FROM original_orders
  WHERE user_id % 16 = 0;
  ```

- **增量同步**：如果业务需要不停服，那么在迁移全量数据期间产生的新数据，需要通过监听数据库的 `binlog` 等方式，实时同步到新的分片表中。

分库的迁移

- **准备阶段**：搭建8个新库。
- **双写阶段**：修改代码，写操作同时写入老库(4个)和新库(8个)。读操作依然走老库。
- **数据补录**：用脚本将老库的历史数据，按新规则计算后导入新库。为保证一致性，导入时发现新库已有数据（由双写产生），则以新库为准或跳过。
- **校验切读**：对数据一致性做校验。
- **流量切换**：将读流量逐步切到新库，观察系统情况。
- **下线阶段**：确认无误后，下线双写逻辑，停用老库。

## 应用层改造

在数据访问层（DAO）编写代码，根据分片键计算出数据应该去哪个表。

借助中间件使用

- **ShardingSphere**：目前最流行的开源方案，提供了JDBC驱动层（Sharding-JDBC，适合Java应用）和代理层（Sharding-Proxy，适合多语言或异构系统）两种接入方式。
- **MyCat**：经典的数据库中间件，独立部署为代理服务，对所有语言透明。

## 监控与维护

- **性能监控**：监控各分片的连接数、QPS、磁盘IO等，确保负载均衡 。
- **扩容预案**：业务会持续增长，当现有分片不够时，需要提前规划扩容方案。对于哈希分片，可以考虑**翻倍扩容**（由16张表扩到32张），这样只需迁移约一半的数据，或者从一开始就设计成**一致性哈希**
