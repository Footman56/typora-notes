# 确保消息不丢失

可能丢失的三种情况：生产者到rabbitmq服务器、rabbitmq消息持久化、消息者拉起消息

## confirm 消息确认机制（生产者）

Confirm 模式是 RabbitMQ 提供的一种消息可靠性保障机制。当生产者通过 Confirm 模式发送消息时，它会等待 RabbitMQ的确认，确保消息已经被正确地投递到了指定的 Exchange 中。
消息正确投递到 queue 时，会返回 ack。
消息没有正确投递到 queue 时，会返回 nack。如果 exchange 没有绑定 queue，也会出现消息丢失。

## 消息持久化

消息存储到磁盘

## ACK 确认机制

成功处理消息后，发送ACK 给rabbitmq, 告知消息可以移除。



# 确保消息不会重复消费

核心是从业务层保证，而不是MQ去保证

1. 数据库使用唯一键约束 如果插入失败就说明重复处理
2. 使用乐观锁：发送消息时带上字段的版本号，消息时检查字段是否有变化，没变化就消费
3. 消费开始时检查是否已经存在记录，是的话就不消费
4. 消息开始时插入mq消费表，如果插入成功就处理业务，否则就是已经消费过啦
4. 高并发情况：lua 脚本 redis 去重  SEKNX  指令 并设置超时时间
4. 数据库状态机：如果操作有状态的话，就通过状态来判断



# 实现消息持久化

## 消息持久化

发送消息时将 deliver_mode = 2

## 队列持久化

经典队列将durable 设置为true，仲裁队列、流式队列 默认必须持久化

普通队列：

+ 先进先出 
+ 竞争消费
+ 多个消费者负载均衡

死信队列：

+ 被拒绝
+ 过期
+ 队列满

延迟队列：

+ 正常队列设置TTL, 过期后进入死信队列

生产环境通常开启持久化、手动 ACK，并配合死信队列实现重试机制

## 交换机持久化

将交换机的durable 设置为true.

# 死信队列

产生死信队列的情况

+ 消费被拒绝
+ 消息达到预计的过期时间没有被消费
+ 消息由于队列达到最大长度被丢弃



给正常队列设置三个参数

1. dead-letter-exchange：指定当前队列对应的死信队列
2. dead-letter-routing-key：指定消息转入死信队列时的路由键
3. message-ttl： 消息在队列中的过期时间。
