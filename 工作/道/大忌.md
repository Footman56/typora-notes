

**清洗数据时要验证之前的老数据，检查对老数据的影响**

**执行更新操作的时候要保证更新字段具有最小维度，保证不影响其他字段**

**回归测试要保证之前的流程能够跑的通**

```
如果出现数据清洗有问题的时候，那么准确定位到出问题的行（因为哪些sql操作导致出现了问题）
需要备份数据。作为恢复的双保险。有了恢复数据才能尽快恢复成正常的状态
删除掉错误数据
插入备份数据
```

**查看数据要浏览完全，不能只关注自己修改的。**

```
要重点查看修改的部分，整体查看数据全貌。
```

**sql查询出来的结果尽量能够覆盖对象的所有属性，不要出现一些属性为空的情况。属性类型要使用包装类型**

**对外提供的接口 的返回值不能包含枚举值**

在修改sql 的时候，一定要检查xml文件中的变量名称与接口定义的变量名称是否一致

做需求的时候，不要画蛇添足，**就算添加新的功能，也一定要测试是否正确**

在xml 中书写sql for-each 的时候使用的是 item 字段，不是index。一版情况下用不到index

**非必要不要轻易修改服务提供者的名称，接口。如果非要修改的话，需要确定好影响范围，影响范围要查全**

**修改当前接口的入参时候一定要查看接口被使用的情况**

清洗数据的时候不要修改 addtime、modtime。**一定要备份数据，如果数据量比较小的话，一定要记录修改了哪些字段。**

如果清洗数据的时候考虑顺序，如果有顺序要求，一定要按照顺序要求执行

在做需求的时候，要明确修改的位置并且同步改造的位置，可以做修改，但是要保证需求改的是确定的。通过**文档**来记录修改的内容，不要偷偷地进行，需要同步到测试，**需要质量保证方案**。

有重大操作的时候，需要明确流程，**之后按照流程来执行**。

**洗数据的要比较洗之间前和洗之后的数据变化**。准备好回滚策略，并准备好备份数据。要明确清洗耗时，如果过于耗时的话，就可以使用多线程处理。

**开发的时候需要重点关注检查接口返回**

正常业务中查询中需要加上 “is_del=0”这个业务条件

在事务与锁的关系中，如果释放锁之后没有提交事务的话， 另一个线程再次进来的时候可以获取锁，但是事务没有提交导致，查询的数据不是最真实的-》 可以改为手动事务，先提交事务再释放锁

锁超时但是此时事务还没有提交的话，另一个线程可以获取锁，但是因事务没有提交的话，导致查询的仍然是之前的数据



**不要操作常量，修改的话就是全系统都能感知到这种变化，如果使用的话，也是要new 对象，不要直接使用的常量**

不要使用Lists.newArrayList()费时，使用 new ArrayList<>()；



**清洗数据的时候一定要对数据结果进行验证。**

清洗出错之后记录出具体操作的数据，**清洗数据粒度最小化，回滚数据粒度最小化**



Jar 中不能使用@AllArgsConstructor 注解，此注解会生成全部参数的构造方法，但是新增字段的时候会没有新字段导致报错，方法是纯手写，手写的话，就算新增字段，也不会影响之前的业务



不论是会议中还是工作中，**不要轻易去反驳领导的制定的政策**，就算是反驳，反驳的理由也要考虑自己、团队的实际情况。实际情况是否在领导的雷区。



在工作中适当的偷懒可以，但是要有技巧，不能太明显，也不能被领导关注。

工作、日常生活中要小透明：不引人注意，不过分张扬，不碌碌无为。

在重要场合（会议中、汇报中）需要表现的时候，不要怂，不要自卑，勇敢的展示自己
