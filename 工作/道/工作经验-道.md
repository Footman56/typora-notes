## 5、道

```
差劲的系统很难找到修改点，很难找到修改点就会容器出bug
在写业务的时候思考如果添加新的功能之后如何处理
如果你要为这个系统添加新的特性时，现在的结构无法方便的打成目的，就需要修改现有结构
写成人能理解的代码才是优秀的程序员
弄清代码的含义
减少变化类的影响范围
可以将类中变化的部分抽取出来，组成不同的状态，使用状态模式来对变化的部分做同一处理
在写代码的时候要时刻注意自己是在添加新的功能还是在重构
随时随地进行重构
日志是用来排查问题的，要打印关键日志：参数，错误码，原因，关键步骤的执行逻辑
学以致用，多使用Stream流操作
注解和日志不是瞎写的。除非有十分明确地目的才来补充注解或者日志
注解要与代码地方实际意义相符
如果一些配置与程序的运行相关，如果想要替换他们，需要先执行替换方法后去掉配置，避免程序运行时出错
在开发之前要了解需求，确定是哪里需要修改的，哪些范围受影响（这个范围要确定好）。
前端穿过来的数据是不可靠的
```

## 经验

```
要把自己的事情表达好，有什么问题就直接说明
主动去问，不要做被动的等待着
有的时候要补充截图作为证据


将项目中搜索单独做成一个层级，在这个层级能够满足多个参数查询。

在设计与前端交互的model中，最好使用Integer 类型来表示布尔值


使用工具
要发问，对项目有自己的问题，理解

在页面上展示优先级最高的功能
```

**注重数据流转，弄清楚如果使用这些功能**，**关键是不同系统之间的数据交互**

讲解的时候需要 自己输出讲给别人。

在和别人交互的时候，需要确定好数据之间的流程,我们之间如何进行数据交换。

在分组讨论的时候需要指定好分工，谁在做什么事。

**最简单的后续拓展是：给字段增加一个状态。在定义的时候尽量抽象成对象的状态。**

**一些特殊情况处理需要记录日志**



关注产品功能，但是遇到不确定的功能逻辑需要找产品确认（你可以思考，但是你还不能从产品的角度思考问题，你可以多列举情景）。



一旦确定好场景，就按照执行，开发。开发完成之后如果有问题就找产品（如果在开发中有问题一定要找产品确定好这个问题）。



对逻辑进行修改之后，**一定要同步其他人**，避免因为人员认知不一样导致重复修改问题，做了多余工作。



一定要记住自己的开发逻辑。开发前一定要确定好产品逻辑，一旦定好，除非特殊情况，否则不建议修改逻辑。



不应该对有具体明确负责人的事情多嘴。

在做分享、阐述自己做的功能的时候，讲述自己在从项目开发过程中遇到的问题，解决思路，解决方法，开发亮点，后续监控。

**问题排查的时候，在一些关键信息上一定要迅速回复，提高沟通效率**

解决问题的时候，需要解决功能问题，也要解决客户的数据问题

代码开发的时候，主流程里面不应该有过多的设值方法

为了使代码清晰度，绝不应使用派生类型的名称访问基类的静态成员。这样做令人困惑，可能会产生一种幻想，即存在两个不同的静态成员。**子类不要使用父类的静态参数、静态方法**



15 分钟解决问题，如果15分钟不能解决的话，就需要暂时放弃，不能再浪费时间在比较难的问题

返璞归真：要优先解决主要问题。就比如博客开发的时候，主要问题是书写内容，而不是过分关注页面样式。保证书写的内容清晰。



对数据进行划分，属于同一类的数据就应该存储到一起。便于后续查询



与同事对接的时候，一定要明确地说出自己需要的是什么，不要让同事去猜！不要含糊不清！描述需要简洁易懂，



在阐述问题的时候需要直接指明问题在哪里。先说问题，再说原因。
