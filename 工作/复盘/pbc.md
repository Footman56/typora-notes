# 一、前期

对设计文档阅读不充分，没有时间阅读，对需求了解的不够明确。导致后面开发的时候一些细碎的场景把握住。需要对这些场景进行梳理。**积累一些绩效场景，比如审批、权限、流程提交**

前期设计的时候参与度不高，没有参与到表结构设计。对表中的一些字段了解不够，并且字段、对象没有进行统一规定，导致在开发到中期的时候推倒，重新设计对象。**需要用一定时间来做统一字段、对象**

定义接口的时候，定义的太模糊，不够全面。一些不需要前端传过来的值都设计上了，其实是不需要的，一些必填字段没有说明。**适用smart-doc时标注必填字段、去掉一些没有必要的字段。向公司、个人账号、部门等都是可以从上下文中拿到的**

接口定义不够全面这点，如果需求变化，似乎是不可避免的。尽可能的参考ui.**对ui图进行划分，那一部分用那个接口，这部分有哪些动作，这部分返回的数据是怎么样的，怎么请求获取到。**

如果有时间的话，可以对每一个接口建立流程图来表明实现思路，对一些复杂问题，可以和前辈们说一下实现思路。

参考一些历史上写过的功能，如果有一些相似的功能是可以参考的。

# 二、中期

首先掌握时间，掌握时间，掌握时间。一定要给自己的开发留多余的时间。最好是根据每一个接口的优先顺序排期。

**一定要预留自测时间。在开发的时候记录一些关键日志，一些简单的postman可以测试**

**如果遇到需求更改的时候，要有记录，要有记录，要有记录。并且能够找到的影响点**

在遇到变化的时候，不要先着急改代码，先电话沟通好这个需求是否合理，之后找到根源改动点，对症下药

规范一下：先获取数据、处理数据、更新数据

**每天要制定开发计划，并且根据开发计划完成。**

# 三、后期

在解决bug的时候先按照优先级来解决，专心解决每一个bug,可以适当的积攒一些不重要的小bug，最后整体发版。

**最理想的情况是能够一次解决bug**

遇到bug的时候，要及时询问测试复现情景。并对一些情景进行举一反三。
